# CardioVision Deployment Details

This document outlines the deployment strategy for the CardioVision project. It is divided into two sections: a practical, developer-centric deployment guide, and a formalized architectural methodology suitable for inclusion in an academic research paper or technical thesis.

---

## Part 1: Developer Deployment Guide (Standard)

CardioVision uses a decoupled architecture consisting of a React single-page application (SPA) and a Python backend optimized for machine learning inference.

### 1. Architecture Stack
*   **Frontend**: React.js bundled via Vite, styled with Tailwind CSS.
*   **Backend**: Python 3.12+ and FastAPI.
*   **Machine Learning**: OpenCV for image processing, Ultralytics YOLOv8s-seg for object detection and segmentation.
*   **Conversational AI**: Google Gemini 2.5 Flash (via `google-generativeai`).

### 2. Backend Deployment Strategy (Cloud Compute/Render/Railway)
Because the backend relies on heavy ML weights (YOLOv8 models) and OpenCV binaries, it requires a robust computational environment. A containerized or dedicated compute service (like Render Webservice or AWS EC2) is recommended over serverless functions.

**Steps:**
1.  **Environment Variables**: Securely input the following in your host provider:
    *   `GEMINI_API_KEY`: Your Google AI Studio key.
    *   `CORS_ORIGINS`: Provide the production URL of your frontend to prevent cross-origin blocks.
2.  **Startup Command**: The internal server must be bound to the dynamically assigned port provided by the cloud service. Use Uvicorn:
    ```bash
    uvicorn main:app --host 0.0.0.0 --port $PORT
    ```
3.  **Dependencies**: The deployment environment will read `requirements.txt`. Ensure `opencv-python-headless` is used instead of the standard `opencv-python` to prevent the container from searching for desktop UI libraries (GUI dependencies) that Linux servers lack.

### 3. Frontend Deployment Strategy (Vercel/Netlify)
The frontend is built statically and requires no runtime server, making edge CDNs strictly optimal.

**Steps:**
1.  **Environment Variables**: Create a production `.env` equivalent configuration in Vercel to point the React app to your live Python backend. 
    *   *Example*: `VITE_API_URL=https://cardiovision-backend.onrender.com`
2.  **API Connection**: Ensure all fetch/axios requests in `src/utils/api.js` use the base URL from the environment variable rather than `http://localhost:8000`.
3.  **Build Configuration**: 
    *   **Build Command**: `npm run build`
    *   **Output Directory**: `dist`
    *   **Framework Preset**: Vite / React.

## Part 2: Academic & Research Paper Format

*Use the following text as a template for the "Implementation", "System Architecture", or "Deployment Methodology" section of your research paper.*

### Section: System Architecture and Implementation Strategy

The CardioVision diagnostic framework is engineered utilizing a decoupled, client-server topology to isolate computationally intensive deep learning inference from user-interface rendering. This bipartite architecture ensures that high-fidelity visualization mapping—including the render of coordinate-based bounding boxes and alpha-blended segmentation masks—does not negatively impact the browser's main thread performance.

**1. Inference Engine and Backend Infrastructure**
The core computational pipeline is hosted within an asynchronous Python backend running the FastAPI framework. Utilizing an asynchronous gateway prevents thread-blocking during parallel image decoding and the execution of the Ultralytics YOLOv8s-seg model. 

For deployment handling, the system is designed to execute as an isolated, containerized microservice. It natively utilizes `opencv-python-headless` for server-side matrix operations, avoiding the systemic overhead of graphical user interface libraries inherent to standard OpenCV builds. Furthermore, the backend acts as a secure middleware layer to integrate a large language model (LLM)—specifically Google Gemini 2.5 Flash—to power the "CardioVision AI Assistant." The implementation utilizes stringent system-level prompt engineering to strictly bound the conversational scope to cardiovascular health information, explicitly rejecting out-of-domain queries to maintain clinical context validity.

**2. Client-Side Rendering and Edge Deployment**
The client-facing interface operates as a dynamic Single Page Application (SPA), constructed utilizing React and optimized through the Vite build tool. The interface utilizes stateless rendering mechanisms to visually represent JSON payloads generated by the inference engine, actively compositing base64-encoded segmentation masks and heatmaps directly over source angiograms. 

For production environments, the frontend application is compiled into statically served assets dispersed across an Edge Content Delivery Network (CDN) (e.g., Vercel). By isolating the presentation layer to an edge network, the system dramatically reduces global physical latency for asset delivery, allowing the primary compute costs and scaling mechanisms to remain isolated exclusively to the backend inference container. Communication between the edge-served client and the computational backend is facilitated over stateless HTTP/REST requests structured with strict Cross-Origin Resource Sharing (CORS) permissions.
